ii. The Inventory

What just got installed.

The kit is intentionally small. Four pieces, each readable in under five minutes. Open the files yourself — they are plain Markdown and JSON, sitting in ~/.claude/.

File · 01 ~/.claude/CLAUDE.md Read on every session.
Sets the tone, the rules,
and the auto-memory protocol.

Identity & standards

The global system prompt Claude reads on every session. It sets the tone (terse, direct), the language you reply in, the "never assume" rule, and the auto-memory protocol that lets the assistant remember decisions across days.

What's inside
  • Identity block — who Claude is when talking to you.
  • Universal coding principles (Karpathy guidelines).
  • "What done means" — checklist for non-trivial work.
  • Auto-memory section — how the assistant remembers across sessions.
File · 02 ~/.claude/skills/ Five hand-picked.
Activate by keyword.
No bloat.

Five skills, hand-picked

Skills are reusable instructions that activate by keyword. You got five: gh-issues, graphify, impeccable, project-prime, ui-ux-pro-max.

Inventory
  • gh-issues — fetch and triage GitHub issues, spawn fixes.
  • graphify — input → knowledge graph → HTML report.
  • impeccable — distinctive frontend code, no AI-slop aesthetics.
  • project-prime — 10-phase framework before starting anything new.
  • ui-ux-pro-max — UI/UX design intelligence with curated styles.
File · 03 ~/.claude/settings.json Plugins declared.
Marketplaces registered.
MCPs opt-in.

Plugins & MCPs declared

The merged settings file enables nine plugins and any MCP servers you opted into. Marketplaces are registered so /plugin install resolves without manual setup on a fresh machine.

Where to look

See the plugins.md and mcps.md catalogs at the repo root for the full list and verified install commands.

File · 04 ~/.claude/projects/<slug>/memory/ An empty index.
Future-Claude fills it in.
Loads on every start.

Auto-memory seed

A starter MEMORY.md waits for the assistant to fill it in. As you work, durable facts get written here and loaded into the context of future sessions.

Lifecycle
  • Read at session start.
  • Written to when you say "remember X" or after non-obvious feedback.
  • Pruned during /close handoffs.
Marginalia · Backup

Your previous ~/.claude/ was copied to ~/.claude/.backup-<timestamp>/ before the install ran. If anything feels wrong, restore is a single directory copy.

iii. The Centerpiece

A day with Claude Code.

Read this in order the first time. Skim it later. Everything that follows assumes you ran claude in a project directory and you have an active session.

What follows is the loop you'll fall into within a week: open a session, declare intent, let Claude verify and propose, then either step into plan mode or work directly. The sections below are the moves inside that loop.

Starting a session

Open a terminal in the project root and type:

terminal 
cd path/to/your/project
claude

On the first message, Claude loads, in order: your global ~/.claude/CLAUDE.md, the project's CLAUDE.md if it exists, and the auto-memory index for the current working directory. That stack of files is the context it reasons from.

Plan mode vs. direct

Claude Code has two operating modes. Choose deliberately.

  • Direct. Claude reads, edits, and runs tools immediately. Use for small, low-risk edits where you'd rather iterate than design.
  • Plan. Claude proposes a plan and waits for approval before touching anything. Use when the change touches more than two files, introduces a dependency, or modifies infrastructure.

Trigger plan mode by pressing Shift+Tab at any prompt, or by asking for it: "make a plan first, don't edit yet." Or use /plan from the feature-dev plugin for the full guided workflow.

Three situations you'll meet this week

i

Implementing a feature

You have a feature request. You don't yet know the right shape.

Open Claude, paste the request, and say: "use spec-driven development." The agent-skills:spec skill kicks in — it interviews you, writes a structured spec, then asks for approval before breaking it into tasks. From there: /plan → /build → /test → /review → /ship.

prompt 
"Add CSV export to the orders page. Use spec-driven development —
write the spec first, get my approval, then plan tasks."
ii

Debugging a failing test

A test is red and you don't know why. Don't ask "fix it" — ask for root cause.

The agent-skills:debugging-and-error-recovery skill enforces the discipline: read the failure, form a hypothesis, prove it with a probe, then write the fix. The Prove-It pattern from agent-skills:test-driven-development requires a test that would have caught the bug before the fix lands.

prompt 
"checkout_test.py::test_discount_applies is failing. Walk me
through root cause. Don't change code until I confirm the
hypothesis."
iii

Reviewing before merge

You have a branch ready. You want a second pair of eyes.

Run /review — the agent-skills:review skill scores the diff on correctness, readability, architecture, security, and performance. For UI-touching diffs, pair with the agent-skills:browser-testing-with-devtools skill, which drives Chrome DevTools MCP to verify the change actually renders.

MCP servers in practice

Three MCP servers carry their weight on almost every project. The others are situational.

context7 — fresh docs, not training data

When you ask Claude about a library, the answer in its training set may be 18 months stale. The context7 server fetches the live docs and cites them.

prompt 
"Show me how to configure middleware in Hono v4. Use context7
to verify — I want the current API, not what you remember."

claude-mem — cross-session memory you can search

Auto-memory is the fast path. claude-mem is the deep archive — every observation from every session, queryable.

prompt 
"Did we already solve the duplicate-row bug in the importer?
Search claude-mem before re-investigating."

sequential-thinking — long reasoning chains

For problems that need stepwise reasoning with revision (design tradeoffs, multi-system debugging), this server gives Claude a scratch pad it can branch and edit.

prompt 
"We need to decide between SQS and Kafka for this pipeline.
Use sequential-thinking to lay out 3 alternatives, then
recommend one."

Hooks — automation without prompting

A hook is a shell command Claude Code runs on its own when an event fires. The kit ships one optional example: the Stop hook, which runs whenever Claude finishes responding to you.

Why a Stop hook matters: Claude can leave the session idle for minutes while it works. The hook can play a notification sound or flash a toast, so you don't sit watching the terminal.

settings.json 
{
  "hooks": {
    "Stop": [
      {
        "matcher": ".*",
        "hooks": [
          { "type": "command", "command": "pwsh ~/.claude/hooks/notify-stop.ps1" }
        ]
      }
    ]
  }
}

Customize by editing the script. On macOS, swap the command for osascript -e 'display notification ...'; on Linux, notify-send. Anything that runs in your shell will run here.

Auto-memory — how the assistant remembers

The kit configures a file-based memory at ~/.claude/projects/<cwd-slug>/memory/. Each remembered fact is a Markdown file with frontmatter (name, description, type). A flat MEMORY.md indexes them — and is loaded into every session automatically.

To save a memory, just say so:

prompt 
"Remember: integration tests in this repo must hit a real
Postgres, never a mock. We got burned by mock/prod drift
last quarter."

To recall, just reference what you want. Claude will read the relevant memory file before answering. Memory is verified against current code on use, not blindly trusted — stale memories get flagged or rewritten.

To forget something: "forget the memory about mock testing — that policy changed."

iv. Make It Yours

The kit is opinionated, not precious.

Every file is editable. Three customization patterns cover 90% of what you'll want to change.

Treat the kit as scaffolding. The opinions in it are mine and the contributors'; yours may differ within a week. The cost of changing any of it is one file edit.

Edit the global CLAUDE.md

Open ~/.claude/CLAUDE.md in any editor. Add a section for your stack, your test runner, your forbidden patterns. Anything you write here is read on every session, in every project.

A useful rule of thumb: if you correct Claude on the same thing twice, write it into the global file.

Add your own skill

Skills are folders under ~/.claude/skills/. Each contains a SKILL.md with YAML frontmatter that declares when it activates.

SKILL.md 
---
name: my-skill
description: When you say "deploy to staging" or
  "promote to prod", use this skill to run the release checklist.
---

# Release checklist

1. Run `pnpm test` and confirm green.
2. Bump version in `package.json`.
3. ...

Save, restart Claude Code, and the skill becomes available. See the plugin-dev:skill-development guide if you want to ship it as a plugin.

Project-level CLAUDE.md

For repo-specific rules — naming conventions, the deploy command, a known-flaky test to skip — drop a CLAUDE.md at the project root. It's appended to the global file when Claude starts a session in that directory.

The kit ships a project-CLAUDE.md.template in templates/ as a starting point.

Marginalia · Precedence

If the project file contradicts the global, the project wins — and Claude is told to surface the contradiction back to you in the same response.

v. Errata

Five things that bite people in week one.

None of them are fatal. Each has a one-line fix.

What follows are five entries pulled from real install dry-runs. Each is filed the way a print magazine files an erratum: the symptom on the left, the correction below. None of them require a reinstall.

Skills don't activate after install.

Claude Code only scans ~/.claude/skills/ at startup. If you installed while a session was running, the new skills aren't loaded.

Fix. Exit and restart the session. /clear won't do it — you need a fresh process.

Permission prompts on every tool call.

Out of the box, Claude Code asks before every non-trivial tool call. Useful at first, noisy after week one.

Fix. Run the fewer-permission-prompts skill — it scans your transcripts and writes a prioritized allowlist into .claude/settings.json. Or edit the permissions block in ~/.claude/settings.json directly.

An MCP server is stuck on "connecting".

Run claude mcp logs <server-name> to see what the server printed on stderr. Common causes:

  • Missing API key in your shell environment.
  • Network blocked (corporate VPN, DNS).
  • The underlying npx package failed to fetch — try npx -y <package> manually.

Remove and re-add with claude mcp remove <server> then the original claude mcp add command.

Auto-memory isn't saving.

The directory at ~/.claude/projects/<cwd-slug>/memory/ must exist before Claude can write to it. The installer creates it, but if you started a session in a directory the installer didn't see, it'll be missing.

Fix. Ask Claude: "create my memory directory and seed MEMORY.md." It will run the right commands.

You want to roll the install back.

The installer saved your previous ~/.claude/ to ~/.claude/.backup-<timestamp>/. To restore, copy the backup back over the live directory:

terminal 
# macOS / Linux
rm -rf ~/.claude/CLAUDE.md ~/.claude/skills
cp -R ~/.claude/.backup-<timestamp>/* ~/.claude/

# Windows PowerShell
Remove-Item -Recurse "$env:USERPROFILE\.claude\CLAUDE.md","$env:USERPROFILE\.claude\skills"
Copy-Item -Recurse "$env:USERPROFILE\.claude\.backup-<timestamp>\*" "$env:USERPROFILE\.claude\"

Nothing else from the kit persists outside that directory, so this is a complete uninstall.
